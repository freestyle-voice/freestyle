## Long-term Knowledge

### Architecture

<!-- lore:019eda8f-2bf4-78ec-b979-8852b205f3dc -->
* **Cleanup intensity/prompt resolution: resolveBaseCleanupPrompt + preset fallback**: \`prompts.ts\` uses \`resolveBaseCleanupPrompt(intensity: CleanupIntensity, customPrompt?: string): string\` to select the active system prompt. For \`"custom"\`: uses trimmed user prompt if non-empty, else falls back to \`CLEANUP\_PRESET\_PROMPTS.low\`. For other intensities: returns \`CLEANUP\_PRESET\_PROMPTS\[intensity]\` directly. \`buildRewritePrompt\` accepts \`intensity\` and \`customPrompt\` options and delegates to this resolver (default intensity: \`"low"\`). Preset bodies live in \`@freestyle/validations\` (\`cleanup-presets.ts\`), not inline in \`prompts.ts\` — the old \`UNIFIED\_REWRITE\_SYSTEM\` constant was removed. \`DEFAULT\_CLEANUP\_INTENSITY = "low"\` and \`CLEANUP\_CUSTOM\_PROMPT\_MAX = 20000\` are defined in \`validations/settings.ts\`.

<!-- lore:019ee543-2f96-7851-af06-01795ab451f4 -->
* **MCP server tool inventory and gap analysis vs HTTP API**: MCP server at \`apps/server/src/routes/mcp.ts\` (mounted \`/mcp\`), server name \`"freestyle"\` v\`"0.0.2"\`. Uses \`StreamableHTTPTransport\`. Tools call \`call(app, method, path, body)\` via in-process \`app.request()\`. Currently 11 tools covering formats (CRUD), dictionary (CRUD), history (list only). Vocabulary is entirely absent. Gaps within exposed domains: Formats missing \`GET /match\`, \`POST /reset\`; Dictionary missing \`GET /all\`, \`POST /export\`, \`POST /import\`; History missing \`GET /stats\`, \`GET /:id\`, \`DELETE /:id\`, \`DELETE /\`. Settings, vocabulary, post-process, transcribe, models, keys, whisper, mlx-asr are fully unexposed.

### Decision

<!-- lore:019ee543-2f99-78f5-9d1f-89c1fe8c085e -->
* **MCP redesign: 15-tool customization-scoped set with vocabulary added and single-item views removed**: Chose 15-tool MCP set over original 11 to cover all customization domains. Changes: ADD \`vocab\_list\`, \`vocab\_create\`, \`vocab\_update\`, \`vocab\_delete\` (vocabulary was entirely missing — biggest gap); ADD \`format\_match\` (calls \`GET /match?context=\`, returns matching format or null); ADD \`cleanup\_get\` (read-only settings fetch). REMOVE \`dict\_view\` and \`format\_view\` — redundant because \`\*\_list\` returns full rows (\`SELECT \*\`). History stays read-only (\`history\_list\` only). Rationale: agent misuse risk when dict vs vocabulary descriptions are ambiguous; single-item views add no value when list returns complete data.

### Gotcha

<!-- lore:019eda7f-9b5d-78fb-8329-83f90a0468dc -->
* **Electron loads @freestyle/server from compiled dist, not TS source — server changes require rebuild + restart**: Trap: editing \`apps/server/\*\*\` TypeScript looks like it should take effect immediately (like renderer hot-reload), but Electron resolves \`@freestyle/server\` via package \`exports\` to \`dist/index.js\` — no \`--conditions=development\` flag is passed, so Node never uses the TS source. Fix: after any server-side change, run \`turbo build\` (or rebuild the server package) then fully quit and restart the Electron app. Renderer changes hot-reload; server changes do not. Symptom: DB has correct values, source code looks right, but behavior matches old logic — stale dist is the culprit.

### Pattern

<!-- lore:019eda79-a6cf-7977-ab47-e17f20c8f6fd -->
* **Custom prompt save: explicit Save button pattern (not debounced autosave)**: The cleanup custom prompt uses an explicit Save button instead of debounced autosave. \`use-models.ts\`: \`setCleanupCustomPrompt\` only updates local state; \`saveCleanupCustomPrompt()\` does the \`PUT /api/settings/cleanup\_custom\_prompt\` and toggles \`savingCustomPrompt\`. \`customPromptDirty\` is derived (current text ≠ last saved baseline). \`cleanup-intensity.tsx\`: Save button (\`variant="ink" size="sm"\`) shown only in Custom mode. Three states: saving (Loader2 spinner), dirty (enabled), clean (disabled). Rationale: debounced autosave had an unmount-flush gap — navigating away within 500ms silently dropped the last edit because \`clearTimeout\` on unmount cancelled the pending PUT without flushing it.

<!-- lore:019ee3de-f95c-7a46-aa90-2f6dee5ba4dc -->
* **Export endpoints: POST /export with body { type } pattern**: Export endpoints use \`POST /export\` with JSON body \`{ type: 'json' }\`. Both dictionary (\`POST /api/dictionary/export\`) and vocabulary (\`POST /api/vocabulary/export\`) follow this pattern. Validated via \`exportSchema\` from \`packages/validations/src/export.ts\`: \`EXPORT\_TYPES = \['json'] as const\`. Routes use \`switch (type)\` with 400 fallback. Vocabulary also has \`POST /import\` using \`INSERT OR IGNORE\`, returning \`{imported, skipped}\`. Dictionary import similarly exists at \`dictionary.ts:155\`.

<!-- lore:019ee543-2f9d-7bbf-927f-c9d108c2928e -->
* **format\_match endpoint: context-aware format selection logic**: \`GET /api/formats/match?context=\<string>\` at \`formats.ts:72\`. Does case-insensitive substring match on pipe-delimited \`app\_pattern\` field. User rules (\`is\_default=0\`) take priority over defaults (\`is\_default=1\`). Returns matching row or \`null\`. \`POST /api/formats/reset\` at \`formats.ts:133\` deletes all non-default rules (\`is\_default=0\`), restoring factory state. Neither endpoint is currently exposed via MCP — \`format\_match\` is planned for addition.

<!-- lore:019eda8f-2bf5-7af9-b71b-9d32dc243ebc -->
* **Settings route key-specific validation pattern**: In \`settings.ts\` route, each settings key has its own validation branch: \`cleanup\_intensity\` is validated via \`cleanupIntensitySchema.safeParse\` (returns 400 \`"Invalid cleanup intensity"\` on failure); \`cleanup\_custom\_prompt\` via \`cleanupCustomPromptSchema.safeParse\` (returns 400 \`"Custom prompt is too long"\` on failure). Schemas live in \`@freestyle/validations/settings\`. \`parseCleanupIntensity()\` coerces arbitrary persisted values to a valid \`CleanupIntensity\` with fallback to \`DEFAULT\_CLEANUP\_INTENSITY\` — use this when reading from DB to handle stale/invalid stored values.

<!-- lore:019ee543-2f9b-73f2-bdd2-2d2956e0d4ca -->
* **Vocabulary vs Dictionary domain distinction in Freestyle**: Dictionary (\`key\`→\`value\`): post-transcription exact string replacement. Vocabulary (\`term\`+\`notes\`): pre-transcription ASR recognition bias — hints the speech model to recognize specific terms. These are distinct pipelines. Schemas in \`packages/validations/src/\`: \`createDictionarySchema\` requires \`key\`+\`value\` (min 1); \`createVocabularySchema\` requires \`term\`, optional \`notes\`. Constants: \`VOCABULARY\_TERM\_MAX=200\`, \`VOCABULARY\_NOTES\_MAX=2000\`, \`VOCABULARY\_IMPORT\_MAX=1000\`. \`GET /vocabulary/all\` returns terms ordered by length DESC (used for ASR injection). Duplicate term on \`POST /vocabulary/\` returns 409.
